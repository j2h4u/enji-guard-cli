# ТЗ: адаптер Enji Guard + MCP-сервер и CLI

> Документ описывает **что** строим и **зачем**, без кода. Технический контракт нижележащего API — в `enji-guard-api.md` (источник истины по эндпоинтам, телам, неймспейсам, auth).
> Версия документа: v1-черновик. Дата: 2026-06-28.

---

## 1. Зачем это и для кого

Enji Guard — сервис, который по расписанию аудитит репозитории и пишет отчёты (безопасность, готовность к AI, тесты, гигиена кода/зависимостей, мёртвый код). Управление через веб-UI неудобно для рутинных и пакетных операций.

**Потребитель инструмента — не человек, а агент-техлид** (LLM-агент, обслуживающий репозитории). Поэтому приоритеты дизайна отличаются от человеческого CLI:

- **экономия контекста** — агент платит токенами за каждый ответ; инструмент отдаёт сигнал, а не шум;
- **структурный I/O** — агент парсит, а не читает; JSON по умолчанию, предсказуемые коды;
- **безопасность автономной работы** — идемпотентность и предпросмотр (`dry-run`) перед записью;
- **сокрытие хаоса API** — единое имя отчёта вместо трёх внутренних неймспейсов; auth-refresh внутри.

Основной сценарий: **смотреть отчёты и заводить по ним исправления силами собственного агента** (Enji-фикс в v1 не используется для починки). Побочная польза: регулировать расписание проверок — пакетно и индивидуально.

## 2. Скоуп

### Входит в v1
- Чтение отчётов в три уровня глубины (обзор → находки → полный отчёт).
- Экспорт нормализованных находок наружу — чтобы чинил **наш** агент.
- Управление расписанием аудитов: пакетно и по одному репо/отчёту.
- Управление email-уведомлениями (галочки «о ручном» / «об автоматическом» завершении).
- Включение/выключение **Enji-автофикса как настройки** (тумблер, без запуска починки сервисом).
- Auth: вход и автоосвежение сессии.

### Не входит в v1 (отложено)
- **`run` / перезапуск / ожидание прогона** (`runs.start/watch/history`) — переаудит после фикса агент проверяет следующим обычным чтением находок. Берём позже.
- Запуск Enji-автофикса как способа починки (PR силами сервиса). В v1 только тумблер вкл/выкл.
- Управление проектами/репозиториями (add/move/unlink), recon, каталог, участники — зеркалим позже.

### Принцип расширяемости
Весь остальной API **может** быть постепенно перенесён в инструмент. Архитектура (см. §3) обязана делать добавление новой операции дешёвым: операция определяется один раз в ядре и проявляется в MCP и CLI по одной строке. Ничего в v1 не должно мешать дозеркалить API позже.

## 3. Архитектура: три слоя

```
enji-core  (адаптер на Python)         ← вся логика
   ├─ MCP-сервер  (ПРИОРИТЕТ)          ← тонкая обёртка
   └─ CLI                              ← тонкая обёртка
```

- **enji-core** — единственное место с логикой: HTTP-транспорт к `fleet.enji.ai`, auth + refresh, нормализация имён, резолв селекторов, идемпотентные операции, dry-run, троттлинг батчей, модель ошибок. Язык — **Python**.
- **MCP-сервер и CLI — тонкие адаптеры**: разбирают вход, вызывают метод ядра, форматируют выход. Никакой бизнес-логики в обёртках.
- Каждая **операция ядра ⇒ один MCP-тул ⇒ одна CLI-команда** (соответствие 1:1). Это правило держит три слоя синхронными.

**Приоритет — MCP** (основной потребитель — агент, которому удобнее структурный tool-I/O, чем парсинг stdout). CLI — вторичная проекция того же ядра для ручных/скриптовых сценариев.

## 4. Нормализация имён (ключевая ценность обёртки)

Один отчёт в нижнем API имеет до **четырёх** разных идентификаторов в зависимости от эндпоинта (см. §3 в `enji-guard-api.md`): route-слаг, `kind` (расписание), `actionKey` (email/логи). Агент не должен про это знать.

Инструмент выставляет **один канон-алиас** на отчёт и резолвит остальное внутри:

| Канон-алиас | Отчёт |
|---|---|
| `security` | Безопасность |
| `ai-readiness` | Готовность к AI |
| `tests` | Тесты |
| `tech-health` | Гигиена кодовой базы |
| `deps` | Гигиена зависимостей |
| `dead-code` | Мёртвый код |
| `recon` | Базовая диагностика |

Алиас — это `enum`: агент физически не может промахнуться мимо неймспейса. Маппинг алиас→(kind, actionKey, route-слаг) живёт в ядре.

---

## 5. Селекторы (как достигается «пакетно И индивидуально»)

Единая грамматика таргетинга и для одной репы, и для всех — это превращает пакетные операции в однострочник.

| Селектор | Значение |
|---|---|
| `<repo>` | одна репа по имени/алиасу (`context-gateway`) |
| `*` | все репозитории |
| `<project>/*` | все репо в проекте (`pets/*`) |
| `--audit security,tests` | подмножество отчётов |
| `--all` | все репо × все отчёты |

Резолв селектора (имя→repoId, `*`→список) — ответственность ядра.

## 6. Пользовательский сценарий (golden path агента-техлида)

1. **Обзор без шума.** Агент запрашивает компактный список: где есть проблемы выше порога, что давно не запускалось. Тратит минимум контекста.
2. **Углубление точечно.** По проблемной паре «репо × отчёт» запрашивает структурный список находок (severity, файл, заголовок, id).
3. **Починка своими силами.** Агент экспортирует находки в нормализованном виде и чинит репозиторий у себя (Enji в починке не участвует). При желании заводит тикет во внешнем трекере из тех же находок.
4. **Проверка позже.** Результат подтверждается следующим обычным чтением находок (в v1 нет явного `run`/watch — переаудит происходит по расписанию).

Вторичные потоки (вне золотого пути): настроить расписание (`schedule`), уведомления (`notify`), включить/выключить Enji-автофикс (`autofix`) — пакетно или индивидуально.

### Иллюстрация потока
```
обзор всех проблемных мест
   → находки по security@context-gateway
      → экспорт находок → [чинит наш агент]
расписание: включить по будням на всех репо разом
```

## 7. Поверхность MCP (приоритет)

Каждая операция — отдельный тул. Описания тулов пишутся **для агента** (он читает их при выборе). Три уровня чтения — три разных тула, чтобы агент сам выбирал глубину и расход контекста.

| Тул | Вход (параметры) | Назначение / выход |
|---|---|---|
| `enji_reports_list` | `selector`, `stale?`, `min_severity?` | Компактный обзор: по строке на «репо×отчёт» — балл, дельта, счётчики severity, «давно не запускали». Пагинируемо. |
| `enji_report_findings` | `repo`, `audit`, `severity?` | Структурный список находок: `id, title, file, severity, hint`. |
| `enji_report_show` | `repo`, `audit`, `format?` | Полный отчёт. Дорого по токенам — только по явному запросу. |
| `enji_findings_export` | `repo`, `audit`, `format` | Находки в нормализованном виде для внешнего фиксера/трекера. |
| `enji_schedule_list` | `selector` | Текущее расписание: enabled, частота, дни, время, пояс. |
| `enji_schedule_set` | `selector`, `freq`, `days`, `time?`, `tz?`, `dry_run?` | Создать/обновить расписание. Идемпотентно. |
| `enji_schedule_toggle` | `selector`, `enabled`, `dry_run?` | Включить/выключить автозапуск без смены остальных полей. |
| `enji_notify_set` | `selector`, `on_manual`, `on_scheduled`, `dry_run?` | Email-галочки завершения (ручного / автоматического). |
| `enji_autofix_toggle` | `selector`, `enabled`, `variant?`, `dry_run?` | Тумблер Enji-автофикса (без запуска починки). |
| `enji_access` | — | План, лимиты, `can_use_schedules`. |
| `enji_auth_status` | — | Залогинен ли, под кем, жива ли сессия. |

Поведенческие требования к MCP-слою:
- **компактность и пагинация** — `enji_reports_list --all` не вываливает всё разом; есть курсор/лимит;
- **idempotent + `dry_run`** на всех write-тулах: при `dry_run` возвращается точный план (какие запросы пойдут) без отправки;
- `audit` — строго enum из канон-алиасов; `severity` — enum (`low|medium|high|critical`);
- ошибки возвращаются структурно (код + сообщение), а не свободным текстом.

## 8. Поверхность CLI (то же 1:1)

Глагольно-ресурсная структура, повторяющая тулы. Примеры вызовов:

```
enji report list --all --stale --min-severity high
enji report findings context-gateway security --severity high,critical
enji report show context-gateway security --format summary
enji findings export context-gateway security --format json

enji schedule list pets/*
enji schedule set --all --freq workdays --days mon-fri --time auto --dry-run
enji schedule disable context-gateway --audit security
enji notify set --all --on-manual --on-scheduled
enji autofix disable --all

enji access
enji auth login        # запускает браузерный OAuth-флоу, сохраняет сессию
enji auth status
```

Конвенции вывода CLI:
- **JSON по умолчанию** (для агента/скрипта), `--pretty` — человекочитаемо; списки — предпочтительно JSONL (стримится/грепается);
- ошибки — структурным JSON в `stderr`;
- осмысленные **exit-коды** (см. §10), чтобы агент ветвился без парсинга.

## 9. Auth и жизненный цикл сессии (требования к ядру)

Из боевого прогона (см. `enji-guard-api.md`, §2):
- API-ключей/PAT нет; целевой auth пока недоступен. Gateway принимает session-cookie на `.enji.ai` и короткоживущий `Authorization: Bearer <access JWT>`.
- Под cookie — короткоживущий access-token, который веб-SPA молча рефрешит. Прямые вызовы рефреш не запускают → при долгой серии запросов прилетает `401 AUTH_INVALID` посреди прогона.

Требования:
1. `auth login` — браузерный Google-OAuth-флоу, сохранение cookie-jar в защищённом месте (keychain/файл с правами).
2. **Автоосвежение сессии**: ядро ловит `401 AUTH_INVALID`, прозрачно обновляет сессию и **ретраит** исходный запрос; обёртки об этом не знают.
3. **Троттлинг/батчинг** пакетных операций (паузы между запросами, разумная конкуренция) — чтобы не упираться ни в rate-limit, ни в протухание токена в середине большого прогона.
4. Никогда не логировать секреты/cookie в открытом виде.
5. Bearer access JWT поддержан как временный credential type, но не считается стабильной заменой будущему официальному API token/PAT.
6. Cookie-refresh отправляется с SPA-like `Origin`, `Referer`, `User-Agent`; это часть временного браузерного хака, не будущего token-auth пути.
7. Docker MCP-сервис запускает background auto-refresh: читает `exp` из `access_token`, refresh-ит заранее, сохраняет новый `access_token` и новый `refresh_token` в writable auth-файл. Refresh-token ротируется, поэтому параллельные refresh защищены lock.

Текущий cookie-refresh — временный совместимый слой до выдачи API token. Когда появится официальный token/PAT, cookie-import, refresh-cookie storage и refresh-retry path удаляются целиком.

## 10. Конвенции вывода, ошибки, коды возврата

**Severity:** `low | medium | high | critical`. `--min-severity` фильтрует по нижней границе.

**Модель ошибки** (единая для MCP и CLI): структура с машинным `code` и человекочитаемым `message`. Минимальный набор кодов:

| code | смысл |
|---|---|
| `AUTH_REQUIRED` | нет валидной сессии (нужен `auth login`) |
| `AUTH_INVALID` | токен протух (ядро должно было отрефрешить; наружу — только если не вышло) |
| `NOT_FOUND` | репо/отчёт не найден |
| `BAD_SELECTOR` | селектор не разрешился ни во что |
| `VALIDATION` | неверные параметры (например, недопустимый `scheduleTimeSource`) |
| `UPSTREAM` | ошибка нижнего API (5xx и пр.) |

**Exit-коды CLI:**

| код | значение |
|---|---|
| `0` | успех, проблем выше порога нет |
| `2` | успех, но найдены проблемы ≥ `--min-severity` (полезно для CI/ветвления агента) |
| `3` | проблема auth |
| `4` | not found / bad selector |
| `1` | прочая ошибка |

## 11. Нефункциональные требования

- **Экономия контекста** как явная цель: дефолтные ответы компактны; полный отчёт — только по запросу; списки пагинируются.
- **Идемпотентность** всех write-операций: повторный прогон с теми же аргументами не создаёт расхождений (PUT нижнего API идемпотентны — это поддерживается естественно).
- **`dry_run`/`--dry-run`** на каждой пишущей операции: возвращает точный план запросов без отправки. Безопасно для автономного агента.
- **Детерминированный маппинг имён** — один источник правды (алиас→идентификаторы) в ядре.
- **Конфиг частоты гибкий**: `freq` и `days` независимы (нижний API не выводит одно из другого) — ядро передаёт оба явно; дефолт времени — `auto` (сервер сам разносит запуски, анти-«эффект толпы»).
- **Логгинг**: stdlib logging, `WARNING` по умолчанию; HTTP-слой логирует operation, method, path, status и latency, но никогда cookie/token, query-string или тело ответа.
- **Контракт нижнего API**: reconstructed OpenAPI живёт в `contracts/enji-openapi.json` и проходит hard gate. Это не официальный контракт Enji, а фиксированная форма reverse-engineering.

## 12. Открытые вопросы для реализатора

1. **Хранение сессии**: keychain ОС vs шифрованный файл — выбрать по платформам, на которых будет жить агент.
2. **Экспорт находок** (`findings export`): фиксировать ли единый нейтральный формат (репо/файл/severity/описание/стабильный id), пригодный и для Linear, и для GitHub Issues, и для собственного агента. Рекомендуется — да, один формат.
3. **Идентификатор находки**: нужен стабильный `id` между прогонами, чтобы агент не чинил одно и то же дважды и мог сопоставлять «закрыто/появилось снова». Проверить, даёт ли нижний API устойчивый id; если нет — синтезировать в ядре (хеш по файлу+правилу+сигнатуре).
4. **Пагинация `reports_list`**: курсор vs лимит-оффсет — на усмотрение, но обязательна для `--all`.
5. **Канон расписания**: подтвердить раскладку `weekly-2x/3x`/`monthly` по датам, если в будущем понадобятся не только будни (для v1 «по будням» этого не требует).
6. **`run` в v2**: заложить место под `runs.start/watch/history`, не ломая контракт v1.
